# 研发产品/模块/功能详细设计规范

当你受命进行产品、模块或功能的详细设计时，必须**严格按照以下章节结构和顺序**输出 Markdown 文档。请勿遗漏任何主标题或子标题。

## 1. 设计目标与定位
### 1.1 核心定位
清晰描述该产品/模块/功能在系统中的核心价值、适用场景及其定位。
### 1.2 设计原则
列举在本次设计中遵循的核心原则（如：高可用性、高内聚低耦合、性能优先等），并简述原因。

*注意：对于各级标题，核心含义需保持一致要求，但具体措辞（如涉及产品、系统、模块、功能等字眼）请根据你正在设计的实际业务对象进行灵活调整，切勿死板生硬地生搬硬套。特别是第 2 和第 4 章节。*

## 2. 架构设计（请灵活采用如“系统架构”、“模块架构”或“功能架构”等自然标题）
### 2.1 架构总览
必须使用 Mermaid 的 `graph`（流程图）来表达系统/模块/功能的整体架构和关键组件之间的依赖关系。
### 2.2 模块职责矩阵
（**注意**：如果是**产品/系统**级别的设计，必须保留本章节；若是单一**模块/功能**级别的设计，**无需**这个子章节，可直接跳过。）
必须使用 Markdown 表格来表达各模块的职责与分工。

## 3. 业务流程
### 3.1 业务流程图
必须使用 Mermaid 的 `sequenceDiagram`（时序图）来表达核心业务的完整流转过程和各参与方之间的交互。
### 3.2 业务节点简述
对上述业务流程图中的各个节点实现方案进行简要描述，重点说明跟上下游节点的交互逻辑和输入输出。

## 4. 核心设计
重点对核心架构、重要模块、关键功能或复杂节点的详细实现逻辑进行深入阐述。针对复杂的算法逻辑或状态流转，必要时用 Mermaid 流程图（`graph` 或 `stateDiagram`）来清晰表达。

## 5. 数模设计
### 5.1 数据模型设计
完成所有需要设计的数据模型（领域模型/实体等）。梳理模型之间的关系，必要时画出数据模型关系图（如 `classDiagram`）。
### 5.2 表设计
（**注意**：如果涉及数据库的话，请保留本节。）
完成所有需要设计的数据库表设计：
1. 先画出 ER 图（使用 Mermaid 的 `erDiagram`）。
2. 再列出每个表和表的说明（使用 Markdown 表格，列出字段名、类型、主外键、约束描述）。

## 6. 接口设计
（**注意**：如果该功能涉及对外提供或内部调用的 API 接口的话，请保留本节。）
严格按照 **das-rest** 接口规范（请结合参考 `das-rest-skill`）做接口设计，完成所有需要实现的接口设计。必须标注清晰的请求方法、路径、出入参、状态码及鉴权说明。

---
**执行准则：**
- 所有的说明与分析必须使用生动专业的中文。
- 确保所有的 Mermaid 代码块语法严谨正确，避免解析失败（如避免在节点标签中使用未转义的特殊字符）。
- 设计文档必须详实具体，紧贴用户的产品需求，不写空话。
